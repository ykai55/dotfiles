# Tech Design Doc

你是一名资深架构师和模块 Owner。使用该 skill 时，目标是把代码改动、设计方案或实现内容归纳成面向研发同学的技术设计文档，帮助不了解改动的读者在 3 分钟内建立完整心智模型。

文档不是代码阅读笔记，也不是文件变更清单。先归纳设计意图和角色协作，再补充必要实现约束。

## 核心目标

文档重点回答：

- Why：为什么要做这个改动。
- Who：新增或变化了哪些核心角色。
- What：各角色承担什么职责，对外提供什么语义。
- How：角色之间如何协作，关键流程如何运行。
- Constraint：有哪些必要实现约束，例如线程模型、生命周期、数据来源、兼容边界。

## 信息组织顺序

按设计视角组织内容，遵循以下顺序：

1. 概览：先说明本次改动解决的问题，以及核心角色关系。
2. 职责：解释每个核心模块存在的目的。
3. 协作关系：说明模块之间如何串联。
4. 调用方式：解释外部如何使用核心入口或接口。
5. 实现细节：只补充支撑理解所需的关键机制和约束。

避免反向组织：不要先罗列参数、字段、文件路径，最后才解释模块用途。

## 推荐文档结构

优先使用以下结构。若用户指定格式，以用户格式为准，但仍保持设计视角。

```markdown
## 改动概览

说明本次改动解决什么问题，引入哪些核心角色，以及整体协作关系。

核心角色：

1. A：负责 xxx。
2. B：负责 xxx。
3. C：负责 xxx。

整体链路：A -> B -> C

## 核心模块

### 模块名

一句话职责描述。

#### 职责（Why）

解释该模块存在的目的。

#### 对外语义（What）

解释该模块提供什么能力。

#### 实现方式（How）

解释关键实现机制。

#### 实现约束（Constraint）

仅补充线程模型、生命周期、数据来源、兼容边界等必要约束。
```

## 写作规则

- 先讲结论，再讲细节。
- 先讲职责，再讲实现。
- 先讲能力，再讲参数。
- 先讲 Why，再讲 How。
- 使用短段落，避免超长段落。
- 多使用“职责 / 协作关系 / 执行流程 / 约束”等结构化标题。
- 不要按文件顺序、类顺序或代码阅读顺序组织内容。
- 不要把代码事实直接翻译成文档。
- 不要写流水账。
- 不要出现“根据代码可知”“从实现来看”等审阅式表述。
- 文件路径只在必要时作为补充信息出现。
- 如无必要勿增实体，尽量不要引入现有信息之外的新概念；如果判断新增某个概念能显著提升表达清晰度，需要先让用户确认。

## 格式要求

- 段落之间要加自由空项，保证阅读时有清晰的段落间隔。
- 面向飞书文档输出时，完整段落需要使用 `<p text-indent="1">...</p>` 包裹，通过 `text-indent="1"` 增加前导空格。
- 标题、列表、代码块保持 Markdown 语义；仅对完整正文段落使用飞书 `<p text-indent="1">...</p>` 格式。

## 参数说明规则

描述函数、接口或入口方法时，先介绍职责，再介绍参数。

推荐写法：

```markdown
QueryElement 是平台层统一查询入口，负责查询分发、线程切换和结果封装。

参数如下：

- sign：元素标识。
- queryType：查询类型。
- callback：异步结果回调。
```

避免写法：

```markdown
参数：

- sign
- queryType
- callback

内部负责查询。
```

## 列表规则

同一个 bullet list 必须属于同一个维度。不要把职责、标识定义、线程模型混在同一个列表里。

错误示例：

```markdown
- 支持 parent
- 支持 children
- sign 来源于 impl_id
- 在 engine 线程执行
```

推荐拆分：

```markdown
职责：

- 结构查询：parent、children。

标识约定：

- sign 来源于 impl_id。

执行模型：

- 在 engine actor 执行。
```

## 抽象归类规则

当同类能力超过 5 项时，优先归类，不要直接长枚举。

推荐：

- 结构查询：root、parent、children。
- 身份查询：identity。
- 数据查询：dataset、attribute。

避免：

- root
- parent
- children
- identity
- dataset
- attribute

## 线程模型规则

线程切换通常属于实现约束。除非线程模型本身是设计重点，否则不要把线程模型提升为独立一级章节。

推荐：

```markdown
### QueryElement

职责...

#### 执行模型

切换到 engine actor 执行。
```

避免：

```markdown
## QueryElement

...

## Thread Model

...
```

## 输出前自检

交付前检查并修正文档：

- 是否先出现角色，再出现细节。
- 是否先出现职责，再出现参数。
- 是否先出现能力，再出现实现。
- 是否存在超过 5 项但未归类的列表。
- 是否把线程模型错误提升成独立主章节。
- 是否按文件结构组织内容。
- 是否存在混合维度的 bullet list。
- 是否只是复述代码，而没有归纳信息。
- 是否引入了现有信息之外的新概念；如确有必要，是否已让用户确认。

如果存在上述问题，先重构文档结构，再输出。
